# 3. Lưu ý quan trọng về Speech-to-Text

OpenRouter chủ yếu được sử dụng cho mô hình ngôn ngữ và không mặc định thay thế hoàn toàn một dịch vụ Speech-to-Text streaming.

Hãy xây dựng một lớp abstraction tên tương tự:

```ts
interface SpeechToTextProvider {
  connect(config: STTConfig): Promise<void>;
  sendAudio(chunk: ArrayBuffer): void;
  disconnect(): Promise<void>;
  onPartialTranscript(callback: TranscriptCallback): void;
  onFinalTranscript(callback: TranscriptCallback): void;
  onError(callback: ErrorCallback): void;
}
```

Kiến trúc phải cho phép thay đổi nhà cung cấp Speech-to-Text mà không cần sửa UI.

Có thể hỗ trợ:

* Web Speech API làm phương án thử nghiệm hoặc fallback.
* Whisper-compatible API.
* Deepgram.
* AssemblyAI.
* Azure Speech.
* Google Speech-to-Text.
* Nhà cung cấp streaming STT khác.

Không được giả định rằng OpenRouter tự động cung cấp streaming Speech-to-Text.

Trong MVP, có thể tạo mock provider để kiểm tra UI, nhưng luồng production phải có interface và vị trí tích hợp STT thực tế rõ ràng.

## 4.4. Nút “Tạo câu trả lời”

Đây là chức năng quan trọng nhất.

Khi người dùng nhấn **“Tạo câu trả lời”**, hệ thống cần:

1. Xác định câu hỏi gần nhất trong transcript.
2. Lấy ngữ cảnh liên quan trước câu hỏi.
3. Kết hợp với hồ sơ cá nhân của người dùng.
4. Kết hợp với system prompt do người dùng thiết lập.
5. Gửi request đến OpenRouter.
6. Stream câu trả lời ngay khi nhận được token đầu tiên.
7. Hiển thị câu trả lời rõ ràng để người dùng có thể đọc trong cuộc họp.

Mục tiêu hiệu năng:

* Phản hồi trực quan sau khi click: dưới 100 ms.
* Hiển thị trạng thái đang tạo câu trả lời ngay lập tức.
* First token lý tưởng: dưới 2,5 giây.
* Câu trả lời ngắn hoàn chỉnh: mục tiêu dưới 5 giây.
* Nếu quá 5 giây, vẫn phải stream nội dung thay vì chờ toàn bộ response.
* Có timeout và thông báo lỗi rõ ràng.

Không gửi toàn bộ transcript dài hàng giờ vào mỗi request.

Hãy xây dựng cơ chế:

* Sliding context window.
* Lấy một số phút hội thoại gần nhất.
* Rolling summary cho phần hội thoại cũ.
* Ưu tiên câu hỏi gần nhất.
* Cho phép người dùng chọn thủ công đoạn transcript làm câu hỏi.
* Giới hạn token đầu vào.
* Hủy request đang chạy khi người dùng tạo câu trả lời mới.

# 10. OpenRouter Integration

Tạo phần Settings cho OpenRouter:

* API key.
* Model.
* Base URL.
* Temperature.
* Max tokens.
* Timeout.
* Context window.
* Ngôn ngữ đầu ra.
* Streaming bật hoặc tắt.
* Test connection.

Model phải có thể thay đổi mà không sửa code.

Tạo service abstraction tương tự:

```ts
interface AIProvider {
  generateAnswer(
    request: GenerateAnswerRequest,
    handlers: StreamHandlers
  ): Promise<void>;

  cancelCurrentRequest(): void;
  testConnection(): Promise<ConnectionTestResult>;
}
```

Request cần chứa:

```ts
type GenerateAnswerRequest = {
  detectedQuestion: string;
  recentTranscript: TranscriptSegment[];
  rollingSummary?: string;
  userProfile?: string;
  customSystemPrompt?: string;
  answerLanguage: string;
  answerStyle: string;
  answerLength: string;
  meetingMode: string;
};
```

Phải hỗ trợ:

* Streaming.
* AbortController.
* Retry có giới hạn.
* Timeout.
* Rate limit error.
* Invalid API key.
* Model unavailable.
* Network error.
* Empty response.
* Response parsing an toàn.

Không log API key.

Không hiển thị API key đầy đủ sau khi lưu.

## Phương án MVP

* Che key trong giao diện.
* Có nút xóa key.
* Giải thích rõ rằng extension chạy phía client nên không thể bảo vệ tuyệt đối API key.

## Phương án production được khuyến nghị

* Sử dụng backend proxy.
* API key OpenRouter nằm trên server.
* Extension sử dụng user token ngắn hạn.
* Có rate limiting.
* Có usage quota.
* Có request validation.
* Có audit log không chứa transcript nhạy cảm.
* Có cơ chế thu hồi token.

Kiến trúc code phải cho phép chuyển từ client API key sang backend proxy mà không cần viết lại toàn bộ ứng dụng.

# 19. Kiến trúc dữ liệu

```ts
type MeetingSession = {

  title: string;

  answerLanguage: string;

  id: string;

  content: string;
  createdAt: number;
  completed?: boolean;
};
```
